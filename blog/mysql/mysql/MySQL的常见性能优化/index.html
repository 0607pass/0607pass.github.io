MySQL的常见性能优化

发表于2025-07-24|mysql
|浏览量:

1、硬件维度

  1. 优化硬件设置（操作系统层参数）或者硬件本身。
  2. 扩容资源（cpu、内存和io等）

2、业务维度

2.1 案例一

  1. 场景和背景：⾃动审核业务数据库主库负载不定时偏⾼（达到正常值 10 倍），慢查询阈值设置为 0.2 秒，但⽆慢查询记录。
  2. 分析：
    1. 观察负载⾼时间点的查询，全部是 select，⼏乎没有 insert，update，delete。
    2. 把 select 查询逐步迁移到 2 台从库执⾏，当迁移⼀条查询时，发现该查询会引起从库负载出现明显增加（达到正常值的 5 倍）
    3. 由此确定这个查询是问题根因之⼀，测试单条 SQL 的执⾏时间在 0.06 秒左右，⽆法进⼊慢⽇志记录（阈值0.2 秒）。这个 SQL 对负载的影响如此⼤主要是执⾏的数量⾮常多（并发⾼）造成，峰值时 QPS值 1000
  3. 优化⽅案
    1. 针对该库负载⾼的 SQL，开发根据业务特征降低了 SQL执⾏频率。具体⽅案为原来 SQL 是 100 个线程每 1秒执⾏⼀次。现在修改为： 如果取到数据的线程，执⾏完成后 sleep 6s，如果没有取到数据的线程，sleep 12s（有时睡眠也是好事）。
    2. 核心：这个优化方案之所以能有效降低数据库负载，核心在于 通过动态调整请求频率，减少了无效的高频查询对数据库的冲击。尤其是避免了大量线程在短时间内重复执行相同查询导致的资源竞争。
    3. 原因和优化思想：原先的100个线程 每秒执行一次查询，无论是否有数据返回，这种模式会持续产生大量请求。 若数据库中有数据，所有线程会频繁竞争读写资源（如锁、CPU、I/O）若无数据，线程仍会重复执行无意义的查询，消耗数据库解析、执行计划的资源。（CPU/IO）
    4. 思想：“有数据稍慢,无数据更慢”降低了无效查询的密度和并发竞争，从而在不改变SQL本身的情况下解决了负载问题。
  4. 优化后效果： 上线后从第⼆天 10 点的数据来看，95% 的线程执⾏完后进⾏ 6s~12s 的 sleep，整体执⾏频率会有⼤幅度降低。

2.2 案例二

  1. 场景和背景：客户调⽤ API 修改价格字段频度增加，数据库峰值 QPS 达 1000。
  2. 分析： 客户操作中，20% 的价格修改前后并⽆变化，但是因为修改还会同时触发表中操作时间字段的修改，数据库仍需要执⾏这些 SQL。伪SQL 如下：UPDATE tb_1 set price=10000.00,price_date=now() WHERE bId=1
  3. 优化⽅案：在此类 SQL 执⾏前判断前后差价，如果相等，不执⾏SQL。
  4. 优化后效果：上线后数据库 QPS 和负载降低 20%。
